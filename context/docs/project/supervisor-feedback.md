> Chronological log of NTNU supervisor directives. One entry per meeting.
>
> **Workflow:**
> 1. Log every directive here verbatim, dated, with the section or scope it touches.
> 2. Generalisable rules (those that apply beyond the immediate section) are mirrored to `evaluation/review/lessons-learned.md` so the writer + reviewer agents enforce them automatically. Cross-link both directions.
> 3. Section-specific calibration is mirrored to the relevant block in `context/outline.md` so the writer sees it at draft time.
> 4. Never delete entries. Strike through and date the correction if a directive is later revised.

**Section-level calibration for the rewrite:**

- **§3.1 (Defining the Task).** Reorder so the task description comes first in the section, before any framing about students or interviews. Drop the phrase "design choices" — frame it as research informing design instead. Replace the unclear word "brief" with concrete language ("the bachelor task offered by Admmit", "Admmit's task description", or similar). Use "their" not "its" when referring to the team. Use "the companies interviewed" (or "the seven companies") rather than the abstract "interviews". Do not claim seven interviews happened in one day; reconstruct the process honestly: one interview at a time, adjusting questions and reflecting between calls; vendor names like Timpex and Opter surfaced gradually through "what tools do you use" questions, with answers describing mixes rather than single-vendor adoption. Strengthen the motivation for two students choosing to interview seven coordinators (currently reads as decision without reason). Do not jump from "seven interviews" to "two design choices were locked" — the connection is missing. Reframe the HITL/multi-tenant origin: "after the interviews, we agreed with what Admmit had set as requirements; Admmit had the requirements, the interviews validated them" — not "regardless of what coordinators said". Drop "not in the interviews" (reads as dismissive / boss-like). Explain multi-tenant in plain language (one deployed system serving several customer companies, each isolated). Drop "they validated but did not introduce" — wrong framing. Drop "so the reader can tell apart" (awkward meta-phrasing). Drop "every design choice traced through" if already said earlier; the repetition up to the HITL paragraph is wasted prose. For the four Trust/control actions: name them inline ("inspect, modify, accept, or reject") rather than referring to "the four actions defined under Trust/control". Simplify the Efficiency sentence. Drop "interview-driven analysis" as unnatural compound. The whole section reads as cross-cutting (sailboat in side-wind) — sequence the argument linearly: task -> why interview -> how interviews unfolded -> what they validated against Admmit's existing requirements -> bridge to method.
- **§2.1 (Resource Scheduling).** Keep the current first sentence and the head-nurse intuition opener — both flagged as good and simple. Replace "combinatorial complexity" with plain step-by-step prose ("the planner has to make a plan so that the traffic coordinator can ..., and ..., and..., which step by step leads to the goal"). Sources may use "combinatorial complexity"; our prose paraphrases.
- **All other sections (Ch 2.2, 2.3, 2.4, 3.2, 3.3, 3.4, 3.5, 3.6, 3.7, and every section yet to write).** The four generalisable rules added 2026-05-04 (open broad / narrow to one message, build understanding not vocabulary, everyday-life intuition before the technical term, no cross-winds) bind these sections too. Existing drafts are marked for a plain-prose pass before final submission; new sections must satisfy the rules at draft time.
